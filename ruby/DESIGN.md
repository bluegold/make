# Ruby Task Runner Design

## 1. 目的

Ruby 版は、Makefile のサブセットを解析して実行するタスクランナーを、Ruby らしい小さなオブジェクトの組み合わせで実装する。

他言語の実装と同じ形に寄せるのではなく、Ruby の強みを使う。

- `Data` で軽量な不変オブジェクトを持つ
- `Set` で探索状態を表す
- `Hash` とブロックでシンプルに組む
- `case ... in` を使って行種別を表現する
- レベル 3 は `Ractor::Port` を使って実行結果の受け渡しを行う

## 2. 期待する CLI

評価スクリプトとの整合を優先し、入口は次の形を想定する。

```bash
ruby/runner <level> <taskfile> [target]
```

- `level`: `level1` / `level2` / `level3`
- `taskfile`: 読み込む定義ファイル
- `target`: 省略時は最初に定義されたターゲット

`tools/evaluate.rb` は `ruby/runner levelX sample.txt` の形で呼ぶ前提にする。

## 3. ディレクトリ構成案

```text
ruby/
├── README.md
├── DESIGN.md
├── runner
├── level1/
│   ├── lib/
│   └── test/
├── level2/
│   ├── lib/
│   └── test/
└── level3/
    ├── lib/
    └── test/
```

### 役割

- `level1/lib`: パーサ、依存解決、直列実行の最小実装
- `level2/lib`: 変数展開を含む実装
- `level3/lib`: 自動変数、タイムスタンプ、並列実行を含む実装
- `levelX/test`: そのレベル専用のユニットテストとサンプルベースの検証

### 再利用方針

level2 と level3 は、level1 で定義したクラスやヘルパーを `require` して使ってよい。

- level1 に、パース結果や依存解決などの共通基盤を置く
- level2 は変数展開層を追加する
- level3 は自動変数、タイムスタンプ、並列実行を追加する

この方針により、上位レベルほどコード量を減らし、差分だけに集中できる。

## 4. データモデル

`Struct` ではなく `Data` を基本にする。理由は、値として扱いたいことと、タスク定義が読み取り中心だから。

```ruby
Task = Data.define(:name, :dependencies, :commands, :source_line)
VariableAssignment = Data.define(:name, :raw_value, :source_line)
ParsedProgram = Data.define(:tasks, :variables, :first_target)
ExecutionContext = Data.define(:target, :dependencies, :auto_vars)
```

### 期待する性質

- `Task#dependencies` と `Task#commands` は配列
- `Task#name` は正規化済みのターゲット名
- `ParsedProgram#variables` は `Hash[String, String]`
- `first_target` は最初に定義されたターゲット名

`Data` を使うことで、パース済みデータと実行時状態を分離しやすくなる。

## 5. パーサ設計

パーサは 1 行ずつ読み、`case ... in` で分類する。

### 行種別

```ruby
case line
in /\A\s*\z/
  :blank
in /\A\s*#/
  :comment
in /\A[ \t]/
  :command
in /\A([^:=\s]+)\s*=\s*(.*)\z/
  :variable
in /\A([^:]+)\s*:\s*(.*)\z/
  :target
else
  :unknown
end
```

### 方針

- 空行とコメントは無視する
- 行頭が空白ならコマンドとして扱う
- `=` を含み、`:` より前に現れる場合は変数定義として扱う
- `target: deps...` はターゲット定義として扱う
- 1 行の中で `:` と `=` が両方ある場合は、Make 風に `=` の位置で判定する

### Ruby らしい点

パーサを巨大な正規表現 1 本にしない。  
行の分類は `case`、抽出は `in` の分解代入で書くと、意図が見えやすい。

## 6. 変数展開

変数展開は 2 段階に分ける。

### 6.1 グラフ正規化時の展開

ターゲット名と依存関係は、パース後に一度だけ正規化する。

- `$(VAR)` を展開する
- 展開結果が複数語なら依存関係として分割する
- 循環変数はここで検知する

### 6.2 実行直前の展開

コマンド文字列は実行直前に展開する。

優先順位は次の通り。

1. 自動変数
2. 環境変数
3. ファイル内変数
4. 空文字

### 再帰展開

`expand(text, scope, visiting = Set.new)` の形にする。

- `visiting` は循環参照検知用
- `scope` は `Hash[String, String]`
- 展開は再帰するが、`Set` でループを止める

### 自動変数

レベル 3 で次を注入する。

- `$@`: 現在のターゲット名
- `$<`: 最初の依存
- `$^`: すべての依存をスペース結合したもの

## 7. 依存解決

依存解決は DFS ベースのトポロジカル順にする。

### 状態管理

- `visited`: 既に完了したノード
- `visiting`: 現在探索中のノード
- `order`: 実行順

ここでは `Set` を使う。

### 挙動

- 依存を先に再帰処理する
- その後で現在ターゲットを `order` に積む
- `visiting` 中に同じターゲットへ戻ったら循環参照エラー

### Ruby での見せ方

`resolve(target, visited:, visiting:)` のように、キーワード引数で状態を明示すると読みやすい。

## 8. 実行モデル

### Level 1

単純な直列実行でよい。

1. 依存順を解決する
2. 各ターゲットのコマンドを順に実行する
3. 実行前に `Executing: ...` を表示する

### Level 2

変数展開を実行直前に挟む。

- コマンドごとに展開
- `@` プレフィックスがあればログ表示を抑制
- 出力順は保持する

### Level 3

次の 3 点を追加する。

- タイムスタンプ比較によるスキップ
- 自動変数の注入
- 依存の並列実行

## 9. タイムスタンプ判定

`needs_update?(target, dependencies)` を分離する。

### ルール

- ターゲットが存在しないなら実行する
- 依存のいずれかがターゲットより新しければ実行する
- 依存が存在しない場合は、ファイル依存として無視する

### Ruby 実装方針

- `File.exist?`
- `File.mtime`
- `File.stat`

この処理は副作用が少ないので、独立したメソッドに切り出す。

## 10. 並列実行

Ruby 版の差別化ポイントはここに置く。

### 方針

`Ractor::Port` を、タスク完了と結果通知の受け渡しに使う。

- 実行要求を投げる
- 完了通知をポートで受ける
- 依存が揃ったタスクだけが次へ進む

### 設計の狙い

- 依存のないタスクを自然に並列化する
- `Task` を `Data` で表し、可能な範囲で shareable にする
- 実行結果の集約口を 1 つに寄せて、ログの責務を分離する

### 実装上の注意

Ractor は共有できない値に厳しい。  
そのため、以下を前提にする。

- タスク定義は早めに freeze する
- 実行メッセージは文字列や配列など、可能な限り単純な形で送る
- 実行本体とログ出力の責務を分ける

`Ractor::Port` の使い勝手が想定より厳しければ、同じインターフェースの内部実装として別の同期機構に差し替えられるようにする。

## 11. コマンド実行

コマンド実行は専用クラスに寄せる。

### 期待する責務

- `Executing:` の表示
- `@` プレフィックスの解釈
- 標準出力と標準エラーの処理
- 異常終了時の終了コード伝播

### 実行方法

直列実行では、出力の順序を保ちやすい方法を優先する。
並列実行では、実行単位ごとの出力を分離し、必要なら後でまとめて流す。

## 12. エラー方針

エラーメッセージは短く、原因が分かる形にする。

- ファイルが無い: `Error: File '...' not found.`
- ターゲットが無い: `Error: Unknown target '...'`
- 循環依存: `Circular dependency detected`
- 循環変数: `Circular variable reference detected`

### 方針

- 例外は内部で使ってもよい
- CLI 層では人間向けメッセージを出して終了する
- サンプル期待値に合わせて文言は安定させる

## 13. レベル別の実装順

### Level 1

- パーサ
- DFS 依存解決
- 直列コマンド実行
- 基本的な CLI とサンプル検証テスト

### Level 2

- 変数テーブル
- 遅延展開
- 循環変数検知
- 変数展開のユニットテスト
- level1 の共通クラス再利用

### Level 3

- 自動変数
- タイムスタンプスキップ
- 並列スケジューラ
- 並列実行とスキップ判定のテスト
- level1 の共通クラス再利用

## 14. この設計で Ruby を選ぶ理由

Ruby は、文字列処理と辞書的データ構造が素直で、DSL っぽい入力を扱うのが得意。

この課題では特に次が相性がよい。

- `Data` によるタスク表現
- `Set` による探索状態
- `case ... in` による行分類
- ブロックによる短い再帰処理

Ruby 版は、他言語よりも「読みやすい小さな実装」を目指す。

## 15. テスト方針

Ruby 版は、各レベルごとにテストを持つ。

### 置き場所

- `ruby/level1/test`
- `ruby/level2/test`
- `ruby/level3/test`

### テストの粒度

- パーサの入力分類
- 依存解決の順序と循環検知
- 変数展開の遅延評価
- 自動変数の注入
- タイムスタンプによる実行スキップ
- 並列実行の完了順と相互干渉の有無

### 実装方針

- 可能なら標準ライブラリの `minitest` を使う
- サンプルファイルをそのまま流して出力比較する統合テストも併設する
- 失敗時に差分が追いやすいよう、アサートは細かめに分ける
